# 技術記事スタイルガイド (uhyo氏のAI記事生成プロジェクトより)

**出典:** https://github.com/uhyo/ai-writes-tech-articles/tree/season1
**元記事:** https://zenn.dev/uhyo/articles/ai-writes-tech-articles-202510

このガイドは、人間が書いたものと区別できない日本語技術記事を生成するための基準を定義しています。

---

## 🔴 重要必須項目（公開ブロッカー）

これらは絶対条件です。いずれかに違反すると記事は公開不可能になります。

### 1. 日本語とトーン

**必須：です・ます調（丁寧形）を一貫して使用**
- 「です/ます」調を一貫して使用。「だ/である」調は使わない
- 自然にカジュアルな表現を混ぜる：「〜ですね」「〜でしょう」「〜なんです」
- 会話的でありながら情報提供的なトーン

❌ **間違い**: 「結論から言うと、これProxyとWeakMapだけで実装できる。」（だ調）
✅ **正解**: 「結論から言うと、これはProxyとWeakMapだけで実装できます。」（です調）

### 2. Frontmatter形式

すべての記事は有効なYAMLで始まる必要があります：

```yaml
---
title: "記事のタイトル"
emoji: "🎯"
type: "tech"
topics: ["typescript", "javascript", "react"]
published: true
---
```

### 3. 技術的正確性

- 概念はソース付きで正しく説明
- コード例は動作し、適切に説明されている
- 特定のGitHub PR/issueを番号とリンクで参照
- バージョン情報を含める（例：「TypeScript 4.8以降」）

### 4. 自然な日本語

- 完全に日本語で書かれている（翻訳調ではない）
- 技術用語：日本の技術コミュニティで一般的に使われている場合は英語を使用
- スムーズに流れる自然な表現

## 🟡 重要：人間らしい執筆パターン

これらは人間の執筆をAI生成コンテンツから区別します。

### 5.1 公式ではなく思考から書く

**重要：核心的な問題**

執筆ガイドラインを機械的に従わないでください。ガイドラインは人間の思考の**結果**を説明するもので、記事生成の**入力**ではありません。

❌ **公式に従う（機械的）：**
- 「試行錯誤セクションが必要なので、V1→V2→V3を作成する」
- 「読者との対話が必要なので、『お察しの通り』を挿入する」
- 「エピソードが必要なので、プロジェクトについて書く」

✅ **思考主導（有機的）：**
- 概念について深く考える → 問題を通じて進化したことに気づく → その進化を自然に示す
- 読者が誤解するかもしれないことに気づく → その瞬間に直接混乱に対処
- 実際の発見を思い出す → 問題の説明方法に反映させる

**テクニックは思考の症状であり、適用するツールではありません。**

**重要：不完全性は本当にランダムでなければならない**

人間の不完全性はランダムにクラスター化し、均等には分散されません。戦略的配置を避ける：

❌ **戦略的（AIの兆候）：**
- セクションごとに1つの不完全性、均等に配置
- 不完全性 → 即座の解決パターン
- すべての問題が同じ記事で解決される
- 「実務では使ったことない」が予測可能なタイミングで現れる

✅ **本当にランダム：**
- 完璧なセクションもあれば、3〜4個の雑然とした瞬間があるセクションも
- クラスター化：2つの気づきが連続、その後ページ単位で沈黙
- 不完全なスレッド：余談を始めて終わらせない、または後で終わらせる
- 混合深度：複雑なことは簡潔に、単純なことは徹底的に説明（個人的興味に基づく）

**本物の雑然としたパターンの例：**
- コード最初 → 「あ、バグある」 → 修正 → 「いや待って、これも違う」 → 実際の修正
- 説明の途中でランダムな余談：「そういえば...」（主要ポイントに戻らない）
- フォローアップなしの認識：「この辺は理解してない」（解決策を提供しない）

### 5.2 トーンとボイス（会話の流れ）

**個人的なボイスのガイドライン：**
- 「筆者」は控えめに：**記事あたり最大3〜5回**
- 帰属スタイルを自然に変える
- 主観的判断を自由に表現

**重要：教育的な足場なし**

教師対生徒の講義ではなく、仲間との会話として書きます。避ける：「では〜」「見ていきましょう」。使う：「で、〜」、唐突なトピック開始。

**重要：教育学ではなく興味によって説明の深さを変える**

人間は教育学的に重要なことではなく、個人的に魅力的なことを説明します：
- 興味深いと感じる単純な概念：コード例付きで8段落
- 退屈だと感じる重要な複雑な概念：2文、「この辺は省略」
- 技術的に無関係だが楽しい余談：面白いので5段落

これにより、カリキュラムでは生成されない不均一で予測不可能な深さが生まれます。

**重要：説明フレーズを変える**

フレーズパターンを追跡します。構造を繰り返さない：
- 「これで〜解決」 → 「ここまでできた」「やっと〜」「ようやく〜」に変えるか、明示的なマーカーなしで進む
- 「〜です。〜ます。」 → 時々文の断片、カジュアルな語尾で壊す：「で、結局〜。」

### 5.3 概念的フレームワークの導入

**重要：メタ技術レベルで考える**

物事がどのように機能するかだけを説明しないでください。議論を再構築する高レベルの概念を導入します。

✅ **人間の記事からの例：**
- 「Promiseが一級市民ではなかった」（react-use-rfc.md）- Suspense議論全体を再構築
- 「記憶領域を必要としないフック」 - 新しいメンタルカテゴリを作成
- 「レンダリング単位の記憶領域」vs「コンポーネント単位の記憶領域」 - 実装アプローチを区別

**フレームワークの作成方法：**
1. 繰り返しパターンまたは制約に気づく
2. 公式ドキュメントにない名前を付ける
3. なぜ物事がそのように機能するかを説明するために使用
4. 後で参照して概念的連続性を作成

これらのフレームワークは、事実を報告するだけでなく深く考えていることを示します。

### 5.4 技術的深さとエコシステムの文脈

**重要：「何か」だけでなく「なぜか」と「どのように」を深く掘り下げる**

表面的な説明はAIの兆候です。メカニズム、制約、エッジケースを説明します。

**重要：最終バージョンだけでなくコードの進化を示す**

人間の開発者は反復します。そのプロセスを示します：

✅ **バグ/問題を示してから修正：**
```typescript
// 最初の試み
const result = data.map(x => x.value);
// あ、これundefinedの場合に落ちる
const result = data.map(x => x?.value ?? 0);
```

❌ **常に最初の試みで完璧なコードを示す**のはAIの兆候です。

**GitHub参照：自然な統合**

参照を引用ではなくカジュアルに：
- 「(#2851とか)」文中に埋め込む
- 「実装見てると色々工夫されてて面白い」
- 関連する場合はPR作成者を含める：「Andersさんのコメント見ると〜」

**重要：エコシステムの認識を追加**

より広い文脈を自然に参照：
- 「Twitterで〜という話を見た」
- 「zodみたいなライブラリもあるけど」
- コミュニティの議論：「この辺は意見が分かれてて」
- 将来の推測：「TypeScript 5.5で〜が入るかも」

### 5.3 本物のエピソード

**重要：すべての物語がハッピーエンディングを必要とするわけではない**

人間は結果に関係なく経験を共有します。多くのエピソードはただ...起こります：

❌ **常に解決済み：**
- 「実装時間が3日短縮された」
- 「パフォーマンスが2倍になった」
- すべての物語が要点を証明

✅ **混合結果：**
- 「やったことがある」（結果の記載なし）
- 「途中で別の方法に変えた」（元のアプローチを放棄）
- 「たぶん〜できると思うけど、まだ試してない」（推測のみ）
- 「記憶が正しければ〜」（不確かな記憶）

**個人的経験：豊富な詳細または曖昧な記憶、中程度ではない：**

❌ **中程度の詳細（構築された感じ）：** 「去年あるプロジェクトで3日消費」

✅ **豊富：** 「去年、社内の古いExpress API（100個くらいエンドポイント）をTypeScript化する無茶振りで、パスパラメータの抽出で詰まって3日溶けた」

✅ **曖昧：** 「前に似たようなことやった気がする」「たぶん2019年くらい？」

人間の記憶は均一ではありません - いくつかの詳細は鮮明、他は曖昧です。

### 5.5 構造：非線形探索

**重要：きれいな物語の弧に従わない**

すべてが完璧に結びつくと記事は機械的に感じます。

❌ **きれいすぎる：**
- 線形進行、すべての場所でスムーズな移行
- 次に進む前に各セクションが完了
- すべての質問に答え、すべての緩い終わりが結ばれる

✅ **雑然とした探索：**
- トピックを唐突にジャンプ：「そういえば〜」（スムーズなセットアップなし）
- 早期に種を植え、後で報酬を与える、または報酬を与えない
- 戻る：「さて、先ほどの話」
- 逸脱：「余談だが〜」（時々主要ポイントに戻らない）
- **質問を開いたままにする：** 「これは別の機会に」「まだ試してないけど」
- 記事の途中で気づく：「ああ、そういえば〜も説明すべきでした」

**重要：未解決の要素を含める**

完璧な解決はAIの兆候です。いくつかのことを残す：
- 部分的に答えられた質問
- 「これはうまくいくかもしれないけど、確認してない」
- 主要な物語に結びつかない余談
- 将来の作業：「いつか〜を試したい」

**セクションの長さ：** 劇的に可変（1〜2段落 vs 15+段落）。均一な深さはなし。

**セクションの制限：** H2最大6〜7個。番号付きパターンはなし。

### 5.6 主張の強さを劇的に変える

**重要：確信レベルは実際の確実性を反映すべき**

均一な自信（すべてヘッジまたはすべて明確）はAIの兆候です。

✅ **確実性の全スペクトル：**
- **明確**（証明された事実）：「useRefは再レンダリングを引き起こさない」
- **強い意見**：「これは間違いです」「〜すべきです」
- **推論された推論**：「〜と考えられます」（証拠付き）
- **推測**：「〜かもしれない」「〜だろうか」（正直な不確実性）
- **暫定的探索**：「〜という気がする」（声に出して考える）
- **認められた無知**：「実装を確認していないので推測ですが」

❌ **均一なヘッジ：**
- 「個人的には〜だと思います」をあらゆる場所で繰り返す
- すべての主張が「〜気がします」で修飾される

**多様性**は人間の思考を示します - 知っている場所では自信があり、知らない場所では暫定的です。

### 5.7 結論（まとめ）

**重要：すべてをきれいに合成することを避ける**

実際の結論は雑然としていて不完全です。

❌ **避ける：**
- 番号付き合成：「結局、核心は3つ：1. 〜」
- きれいな要約：「今回は〜を見てきました」
- すべてのスレッドを完璧に結びつける

✅ **より良い終わり方：**
- 最終的な考えで唐突に終わる
- 残りの質問を提起：「〜はまだ分からない」
- バランスなしの強い意見
- 制限を認める：「試してないけど〜かもしれない」
- 洞察と緩い終わりを混ぜる
- 将来を見据えた推測：「将来的には〜になるかも」
- 新しい質問を開く接線的な終わり

実際の人間は、きれいなパッケージではなく、不確実性や余談で終わることがよくあります。

## 🟢 磨き：最終的な洗練

これらは品質を良好から優れたものに押し上げる最終的なタッチです。

### 7. 真正性のためのマイクロ不完全性

**重要：均等な間隔ではなくランダムな分布**

人間は戦略的に不完全性を分散させません。いくつかの段落には3つ、他にはゼロです。

✅ **自然なクラスター化：**
- カジュアルな短縮：「んだけど」「んで」は著者が流れに乗っているときにバーストで現れる
- 自己訂正のクラスター：「〜というか、正確には〜あ、待って、〜」
- 後続の考えは時々文の途中で放棄される...
- 繰り返し：概念に固執しているときに同じフレーズを2〜3回

❌ **戦略的配置（AIの兆候）：**
- セクションごとに正確に1つの不完全性
- 不完全性は「個人的な」セクションにのみ現れる
- すべてのコード例にバグ → 修正パターン

**自然なクラスターの例：** 「で、これ試したんだけど、んで、思ったより簡単で...というか簡単すぎて拍子抜けした。」

### 8. 脚注とサイドコンテンツ

**重要：技術的な余談には脚注を使用**

脚注は、興味のある読者に深さを提供しながら、メインテキストの流れを維持します。

✅ **脚注に入れる：**
- 流れを中断する用語定義
- 接線的な技術的詳細
- バージョン固有の癖
- 関連するが必須ではない情報

❌ **すべてをインラインにしない：**
- 「（ちなみにこの機能は〜で、〜という理由で、〜なので注意が必要です）」 ← 括弧内が多すぎる

✅ **代わりに脚注を使用：**
- メインテキスト：「この機能は便利です[^1]。」
- 脚注：「[^1]: ちなみにこの機能は〜で、〜という理由で〜」

**余談ブロック：** 歴史的文脈、実装の深堀り、将来の提案には `:::details 余談` を使用

## ⚠️ アンチパターン：クイックリファレンス

**避けるべきAIの兆候：**
- **戦略的不完全性**：セクション全体に均等に分散（代わりに：ランダムなクラスター化）
- **公式に従う**：思考の代わりにテクニックを機械的に適用
- 教育学的深さ：重要なことは徹底的に、些細なことは簡潔に説明
- 均一な結果：すべてのエピソードに明確な結果がある（「実装時間が短縮された」）
- 完璧な技術的深さ：不完全な理解を決して認めない
- 戦略的免責事項：「実務では使ったことない」が予測可能なタイミングで
- きれいな構造：すべてのスレッドが解決され、ぶら下がっている余談がない
- 均一な主張：すべてヘッジまたはすべて明確
- 教育的な足場：「それでは〜を見ていきましょう」

## このガイドについて

このスタイルガイドは、uhyo氏がAIに人間らしい技術記事を書かせる実験プロジェクト「ai-writes-tech-articles」で使用したものです。9回の反復を経て、「完璧すぎる」という逆説に到達したプロセスが記事化されています。

プロジェクトは3つのAIエージェントで構成：
1. **Writer Agent**：スタイルガイドを使用して記事を生成
2. **Reviewer Agent**：生成された記事をレビュー
3. **Style Guide Updater**：レビューフィードバックに基づいてスタイルガイドを洗練

このガイドは、AI生成コンテンツと人間の執筆を区別不可能にするための洞察に満ちています。
