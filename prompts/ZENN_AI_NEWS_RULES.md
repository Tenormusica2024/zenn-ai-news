# Zenn AI News Project Rules

## 📋 プロジェクト概要
海外のAI関連ニュースを自動的に収集・要約し、Zennに投稿するシステム

---

## 🎯 記事選定ルール

### 基本方針
- 幅広い読者層を対象とした記事作成
- AI関連の最新アップデート・トレンド・ニュースを中心に扱う
- 技術的に深すぎず、一般のエンジニア・AI興味層が理解できる内容
- 注目度が高く、話題性のある記事を優先的に選定

### ターゲット読者
- エンジニア（メインターゲット）
- 実際にAI/LLMを使って開発している人
- 新しいツール・API・機能を試したい技術者
- 技術的な詳細に興味がある層

### 記事の粒度
- 一般向けの大きなニュースではなく、エンジニア向けの具体的な機能・アップデート
- 例: 「Claude 4.5リリース」ではなく「Claude 4.5のMCPサポートが強化」
- 例: 「OpenAI発表」ではなく「Codexの新API機能追加」
- 実装例・ユースケースが明確なもの

### 記事の価値提供（最重要）
**🚨 CRITICAL: ただのニュース発表ではなく、注目度の裏付けと実用性を示す**

- **バズ度・注目度の数値的証拠を必須で提示**:
  - Hacker Newsスコア（最低200ポイント以上推奨）
  - Redditのupvotes数（最低1000以上推奨）
  - GitHub Starの急増（具体的な増加数・期間を記載）
  - Twitter/X上でのインプレッション・リツイート数
  - ダウンロード数、npm/pip等のパッケージマネージャーの統計
  - **注目度が数値で証明できないニュースは採用しない**

- **開発者の実際の関心事を反映（補足的に使用）**:
  - コメント欄での議論内容を要約（数値的裏付けの補足として）
  - 「何が便利になったのか」「何が解決されるのか」を具体的に
  - ベンチマーク結果、パフォーマンス比較などの実測値

- **実用的な視点を提供**:
  - 「自分のプロジェクトに使えそうか」の判断材料を提供
  - 料金・制限・対応言語などの実務的な情報
  - 既存ツールとの比較、移行コストの考察
  - 実際の活用シーン・ユースケースの提示

**NG例**: 「〇〇社が新機能をリリースしました。便利です」
**OK例**: 「Hacker Newsで1200ポイント獲得した〇〇の新機能。開発者が特に注目しているのは△△の部分で、既存の方法と比べて3倍速いとのベンチマーク結果が出ています」

### 選定基準

#### 1. コンテンツタイプ（優先度順）
1. 最新アップデート - GPT-5リリース、新モデル発表等
2. トレンド - 話題のAI技術・手法・アプリケーション
3. ニュース - AI業界の動き、企業発表、規制関連
4. 実用事例 - AIの実際の活用例、成功事例

#### 2. ソース優先度
Tier 1（最優先）:
- TechCrunch
- The Verge
- Ars Technica
- VentureBeat AI
- AI公式ブログ（OpenAI, Google AI, Anthropic等）

Tier 2（推奨）:
- MIT Technology Review
- Wired
- Hacker News（高スコア記事）
- Reddit r/MachineLearning（人気投稿）

Tier 3（補足）:
- Medium（AI著名人の記事）
- 企業技術ブログ
- AI系YouTuberの話題

#### 3. 注目度指標（必須・数値化必須）
**🚨 CRITICAL: 注目度が数値で証明できないニュースは不採用**

最低基準（以下のいずれか1つ以上を満たす）:
- Hacker Newsスコア: 200ポイント以上
- Reddit r/MachineLearning upvotes: 1000以上
- Reddit r/programming upvotes: 1000以上
- GitHub Stars: 1週間で+1000以上の急増
- Twitter/X: 主要AI開発者アカウントのツイートで1000インプレッション以上
- npm/PyPI等: 週間ダウンロード数の大幅増加（前週比+50%以上）

推奨基準:
- Hacker Newsスコア: 500ポイント以上
- Reddit upvotes: 2000以上
- GitHub Stars: 5000以上または急激な増加トレンド
- 複数のソースで同時に話題になっている

**記事の新鮮度: 必ず3日以内（絶対条件・鮮度重視）**

#### 4. 記事の分かりやすさ基準
採用基準:
- 専門用語の適切な説明がある
- 具体例・ユースケースが示されている
- 図表・画像で視覚的に理解しやすい
- 一般エンジニアが読んで理解できるレベル

除外基準:
- 数式だらけの学術論文
- 過度に専門的な実装詳細のみ
- 広告・宣伝目的の記事
- 信頼性の低い情報源
- 既に日本語で広く報道済みの話題

## 記事作成ルール

### 人間らしさを重視した記事執筆方針

**最重要原則: AIが書いた記事っぽくしない**

- 機械的な列挙ではなく、会話的なトーン
- 個人的な視点・感想を適度に含める
- 完璧すぎる文章を避け、自然な言い回しを使う
- 絵文字は使用禁止
- 定型文・テンプレート的表現を避ける

### 人間らしい文章の特徴
- 「〜だと思います」「〜かもしれません」など、適度な曖昧さ
- 「個人的には」「興味深いのは」など、書き手の視点
- 「ただし」「一方で」など、自然な接続詞
- 箇条書きばかりでなく、段落で説明
- 完璧な整理ではなく、話の流れを重視

### 🚨 絶対に避けるべきAI臭い表現
**NG例**: 
- 「でもそれこそが価値なんだと思います」← ひねりなしの賞賛的表現は超AI臭い
- 「〜は素晴らしいと思います」
- 「これは画期的です」
- 「非常に重要です」
- 機械的な肯定・称賛だけで終わる文章

**OK例（共感できる軽い懐疑・テクニカルな実体験ベース）**:
- 「正直、『当たり前じゃん』と思った。でも前にコンテキスト管理を疎かにしてレイテンシ3倍になった経験があるから、体系化は重要かも」
- 「理想論だけど、実際はステートフル管理のコストで詰まりそう」
- 「前に似たアプローチでトークン消費が予想外に増えたから、今回はどうだろう」
- 「便利そうだけど、既存パイプラインへの組み込みコストが気になる」

**⚠️ 避けるべき強すぎる批判**:
- 「これは使えない」「意味がない」などの断定的否定
- 「こんなの誰が使うんだ」のような攻撃的な表現
- AI業界では強い語気の批判は少ないため、そぐわない
- 共感できない批判、建設的でない批判は不要

**適切なトーンバランス**:
- 軽い疑問・懐疑はOK：「本当に実用的かな？」「ハードル高そう」
- 自分の経験からの不安はOK：「前に詰まったから心配」
- 期待と不安の両方を書く：「期待はあるけど、〇〇が心配」
- 基本は前向き、ただし盲目的な賞賛は避ける

### 実体験シミュレーションの追加
**自分事として語る（テクニカルな失敗談）**:
- 「前にコンテキスト詰め込みすぎてレイテンシが3倍になった」
- 「同じようなアプローチ試したら、トークン消費が予想外に増えて予算オーバー」
- 「この手法、ステートフル管理が必要で実装コスト高そう」
- 「似たパターンで実装したら、エッジケースの処理で詰まった」

**⚠️ 避けるべきバカっぽい失敗談**:
- 「ドキュメント読んでなくて失敗した」← ただのミス
- 「環境構築で半日溶かした」← 技術的洞察なし
- 「変数名間違えてバグった」← レベルが低い
- 「コピペミスで動かなかった」← 論外

**技術的な悩み・疑問を入れる**:
- 「便利そうだけど、既存のパイプラインにどう組み込むか」
- 「理屈は分かるけど、スケールした時のパフォーマンスが心配」
- 「これ、本番環境でのエラーハンドリングどうするんだろう」
- 「メモリ効率とのトレードオフをどう評価すべきか」

### 楽しさ・エンタメ性の追加
- 技術的な話にユーモアを混ぜる
- 「あるある」ネタで共感を誘う
- 自虐的な表現も適度に入れる
- 読んでいて「クスッ」となる要素

### 🚨 翻訳調・AI翻訳臭さの完全排除
**NG: 翻訳で出した日本語のような文章**

翻訳調の特徴（絶対禁止）:
- 欧米人の感情的表現をそのまま直訳したような文章
- 言葉が綺麗すぎて子供の言葉遣いのように聞こえる
- 「〜を与えています」「〜を巻き起こしています」などの硬い動詞表現
- 「特に注目されているのは」「興味深いのは」の多用
- すべての文が同じトーンで書かれている（温度感が一定）

**OK: エンジニアらしいプロフェッショナルな文体**

基本は淡々とした説明:
- 無理に注目を引く表現を使わない
- 事実を淡々と述べる（「この研究は〜を示した」程度でOK）
- 不要な修飾語を削る
- 技術用語はそのまま使う（変に言い換えない）

ポイントで尖った表現を入れる:
- 本当に重要な部分だけ、エンジニアらしい尖った言い方
- 「これはヤバい」「マジで厄介」など、感情が動いた部分のみ
- 技術的に問題だと思う箇所は、遠慮なく指摘
- ただし、全体の1〜2箇所程度に抑える

**文体の温度差をつける:**
- 9割: 淡々とした事実説明（「〜である」「〜だ」）
- 1割: 感情がこもった尖った表現（「これは危ない」「実装コストエグい」）

**🚨 立場・距離感の適切な設定:**
- **NG**: LLM開発者のような当事者意識が強すぎる表現
- **NG**: 「これはマズい」のような直接的に影響を受ける立場の断定
- **NG**: 映画の主人公のような劇的・感情的な言い回し
- **OK**: 間接的に関係するかもしれない観察者としての適度な関心
- **OK**: 「業界的には重要そうだから追ってる」程度の距離感
- **OK**: 「自分には直接関係ないけど、気になる」というスタンス

**NG例（当事者意識が強すぎる）:**
- 「これはマズい」← LLM開発者でもないのに当事者のような言い方
- 「我々開発者にとって」← 直接的な立場を主張
- 「今すぐ対処が必要だ」← 映画的・劇的な表現

**OK例（適度な距離感）:**
- 「これはやばいんじゃない？」
- 「直接関係ないけど、業界的には気になる動き」
- 「開発者じゃないから実感ないけど、影響ありそう」
- 「個人的には使わないけど、トレンドとしては追っておきたい」

**NG例（翻訳調・子供っぽい）:**
- 「この研究は、AI開発者コミュニティに大きな衝撃を与えています」
- 「特に注目されているのは、その攻撃の『手軽さ』です」
- 「これが意味するのは、『250個の文書を作るのは簡単』という現実です」

**OK例（エンジニアプロ調・適度な距離感）:**
- 「Hacker Newsで563ポイント。それなりに話題になってる」
- 「問題は攻撃が簡単すぎること」
- 「250個なら個人でも作れる。やばいんじゃない？」

```markdown
---

## [記事ごとに完全に異なる見出しを作成 - 例: "SWE-Bench Proの結果が話題に", "Anthropicのポイズニング研究が出た", "Claude 4.5のMCP統合が強化"]

[Hacker News/Reddit/Twitter等での反応を具体的数値で提示]
[開発者コミュニティでの注目理由を説明]

## [記事ごとに完全に異なる見出しを作成 - 例: "新ベンチマークの仕組み", "攻撃手法の詳細", "追加された機能"]

[元記事の要約を会話的に説明。3-4段落程度]

## [記事ごとに完全に異なる見出しを作成 - 例: "Hacker Newsでの議論", "開発者の懸念点", "実装者の反応"]

[コメント欄・議論から見える実際の関心事]
[ベンチマーク・パフォーマンス比較などのデータ]

## [記事ごとに完全に異なる見出しを作成 - 例: "本番環境で使えるか", "実装コストの現実", "既存システムとの統合"]

[自分のプロジェクトで使えそうか]
[既存ツールとの比較、移行コスト]
[料金・制限・対応言語などの実務情報]

## [記事ごとに完全に異なる見出しを作成 - 例: "個人的に気になった点", "懸念事項", "期待と不安"]

[個人的な感想・考察を含める]
```

**🚨 見出しの定型化完全禁止（最重要）:**
- **「注目されている理由」「なぜ注目されているのか」「主な内容」「コミュニティでの議論」などの抽象的・汎用的見出しは使用禁止**
- **記事の具体的な内容に基づいた固有の見出しを作成する**
- **NG例**: "注目されている理由" / "なぜ注目されているのか" → すべての記事で使える汎用表現
- **OK例**: "SWE-Bench Proの結果が話題に" / "$100でChatGPTクローンが作れる時代" → この記事専用の具体的表現
- **NG例**: "主な内容" → どの記事にも当てはまる抽象表現
- **OK例**: "攻撃手法の詳細" / "新ベンチマークの仕組み" → この記事の内容を具体的に示す
- 同じパターンの見出し（"〇〇の理由"、"〇〇の内容"等）も避ける
- 記事タイトルやトピック名を見出しに含めることで具体性を出す

**🎯 最初の見出しは特に重要（読者の興味を引く）:**
- **第一見出しは記事の核心を端的に示し、読者の興味を引く表現にする**
- **NG例**: "概要" / "背景" / "はじめに" → 退屈で汎用的
- **OK例**: "$100でChatGPTクローンが作れる時代" / "GPT-4を超えるベンチマーク結果" → 具体的で興味を引く
- 数字・具体的な成果・驚きの要素を含めると効果的
- 「〜が〇〇になった」「〜で△△を実現」のような具体的表現を推奨
- 最初の見出しで記事の価値提案を明確に示す

**語尾の統一:**
- 記事本文は「ですます調」で統一
- 「〜だ」「〜である」調は使用禁止
- 自然な会話調を保ちつつ、敬語で統一

### 記事作成時の必須ルール
1. 必ず `published: false` で作成
2. 元記事のURL・タイトルを明記
3. 著作権・引用ルールを遵守
4. 機械翻訳そのままではなく、自然な日本語に
5. 技術用語は文脈に応じて解説
6. 絵文字は使用禁止（emojiフィールドのみOK）

### 避けるべき表現
- 「〜について説明します」系の導入
- 「以下のポイントが挙げられます」
- 「〜することができます」の多用
- 完璧に整理された箇条書きのみの構成
- 感情のない機械的な要約
- **「でもそれこそが価値なんだと思います」のような無批判な賞賛**
- **「これは素晴らしいです」だけで終わる浅い評価**

### 記事執筆時のチェックリスト
- [ ] AI臭い賞賛表現（「それこそが価値」等）を削除したか？
- [ ] 実体験シミュレーション（「自分も前に〇〇した」等）を入れたか？
- [ ] 軽い疑問・懐疑的な視点を入れたか？（強すぎる批判は避ける）
- [ ] トーンは前向きだが、盲目的な賞賛になっていないか？
- [ ] ユーモアや「あるある」ネタを入れたか？
- [ ] 「完璧すぎる文章」を崩したか？
- [ ] 共感できない批判や攻撃的な表現を削除したか？
