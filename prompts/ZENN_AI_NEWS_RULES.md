# Zenn AI News Project Rules

## 📋 プロジェクト概要

**目的:** 海外のAI関連ニュースを自動的に収集・要約し、Zennに投稿するシステム

**対象読者:** エンジニア、AI/LLM開発者、新しいツール・API・機能を試したい技術者

**記事種別:**
1. **通常のAIニュース記事** - 従来通りの選定基準で幅広いトピックをカバー
2. **AI系新ツール・新コンテンツ・アップデート専用記事** - ポジティブなAI系新規リリース・アップデートに特化

---

## 🛠️ AI系新ツール・新コンテンツ・アップデート専用記事ルール

### 1. 記事選定基準（新ツール・新コンテンツ・アップデート特化）

**対象となるコンテンツ:**
- **新規AIツール・サービスのリリース**（完全新規製品・サービス）
- **新規AIモデル・データセット・ライブラリの公開**
- **既存AIツール・サービスの機能追加・改善**
- **新バージョンリリース**（メジャー/マイナー問わず）
- **API・SDK・プラグインの新機能追加**
- **パフォーマンス改善・制限緩和のアップデート**
- **新規AI関連コンテンツ**（チュートリアル・ガイド・ドキュメント等）

**ポジティブ基準:**
- ツール利用者にとって明確なメリットがある
- 実用性が向上する機能追加
- コストパフォーマンスが改善される
- 開発者の作業効率を向上させる

**除外基準:**
- 単なるバグフィックスのみ（重大な不具合修正は除く）
- UI/UXの微細な変更のみ
- 既に広く報道済みの大型アップデート
- ネガティブな変更（値上げ、機能削減等）

### 2. ソースコード掲載ルール

#### 2-1. ソースコード選定基準（最重要）

**🚨 CRITICAL: ソースコードは「ツール導入時に最も重要な情報」のみ掲載**

**掲載すべきソースコード:**
- ツールの基本的な使い方を示すコード例
- 新機能を実際に使用する最小限のコード
- 公式ドキュメントから引用した推奨実装例
- API認証・初期設定に必須のコード

**❌ 掲載してはならないコード:**
- 「誰でもわかりきっている」基本的なコード
  - 例: `import requests` だけのコード
  - 例: 単なる変数定義のみ
  - 例: Hello World的な説明不要のコード
- ツール固有でない汎用的なコード
- 複雑すぎて記事の範囲を超えるコード
- セキュリティリスクを含むコード（APIキーのハードコード等）

#### 2-2. ソースコード掲載例

**✅ 良い例:**
```python
# Claude API の新しいメッセージング機能を使用
import anthropic

client = anthropic.Anthropic(api_key="YOUR_API_KEY")
message = client.messages.create(
    model="claude-3-5-sonnet-20250110",
    max_tokens=1024,
    messages=[
        {"role": "user", "content": "Hello, Claude"}
    ]
)
print(message.content)
```

**❌ 悪い例（明らかすぎる）:**
```python
# これだけでは情報源として無意味
import anthropic
```

#### 2-3. ソースコードの記載方法

**必須要件:**
- コードブロックにシンタックスハイライト指定（```python, ```javascript 等）
- コードの前に簡潔な説明文を追加
- 重要な部分にコメントを付ける
- 公式ドキュメントのURLを併記

**掲載箇所:**
- 記事本文の該当機能説明セクション内
- 「実際の使い方」「実装例」等の見出し配下
- 自然な流れで挿入（唐突な掲載は避ける）

### 3. 画像掲載ルール

#### 3-1. 画像選定基準

**🚨 CRITICAL: 画像は「ツールに関係する有用な情報源」のみ掲載**

**掲載すべき画像:**
- 新機能のUI/インターフェース画面
- 出力結果の比較画像（改善前後）
- アーキテクチャ図・フロー図
- 公式ブログ・ドキュメントの図表
- ベンチマーク結果のグラフ

**❌ 掲載してはならない画像:**
- 「誰でもわかりきっている」一般的な画像
  - 例: 企業ロゴのみ
  - 例: 汎用的なAIイメージ画像
  - 例: 情報量のない装飾画像
- ツール固有の情報がない画像
- 解像度が低く視認性の悪い画像
- 著作権上問題がある画像

#### 3-2. 画像検索・選定プロセス

**必須手順:**
1. **WebSearchでツール名 + "screenshot" / "interface" / "demo" を検索**
2. **公式サイト・公式ブログの画像を最優先**
3. **技術ブログ・レビュー記事の画像を次点**
4. **Twitter/X等のSNS投稿画像は補足的に使用**

**画像選定の判断基準:**
- その画像を見ることでツールへの理解が深まるか？
- テキストだけでは伝わらない情報を含んでいるか？
- 読者が「見てよかった」と思える価値があるか？

#### 3-3. 画像アップロード・掲載方法

**アップロード手順:**
1. WebSearchで見つけた画像URLを確認
2. 画像をダウンロード（適切な解像度で）
3. Zenn記事ディレクトリの`images/`フォルダに保存
4. ファイル名は`[ツール名]_[内容]_[日付].png`形式

**Markdown記載例:**
```markdown
![新機能のインターフェース](images/claude_new_interface_20250110.png)
*Claude 3.5 Sonnetの新しいメッセージング画面*
```

**掲載箇所（1-2箇所）:**
- 記事の冒頭付近（ツール全体像を示す画像）
- 新機能説明セクション内（該当機能のUI画像）
- 自然な文脈で挿入（唐突な掲載は避ける）

#### 3-4. 画像の著作権・引用

**必須ルール:**
- 公式サイト・公式ブログの画像を優先使用
- 引用元URLを画像下部に明記
- 改変が必要な場合は最小限に留める
- 商用利用不可の画像は使用しない

### 5. 通常記事との共通ルール

**AI系新ツール・新コンテンツ・アップデート専用記事でも以下は完全遵守:**
- 記事作成ルール（人間らしさ、AI臭い表現回避）
- 翻訳調・AI翻訳臭さの完全排除
- 見出しの定型化完全禁止
- 語尾の統一（ですます調）
- 記事執筆時のチェックリスト全項目

**差分:**
- ソースコード1-2箇所掲載（必須）
- 画像1-2箇所掲載（必須）
- ポジティブな新規リリース・アップデートに特化
- ツール導入実用性を重視
- 新規ツール・新規コンテンツも対象に含む

## 🎯 記事選定ルール

### 1. 基本方針

**記事の特性:**
- 幅広い読者層を対象とした記事作成
- AI関連の最新アップデート・トレンド・ニュースを中心に扱う
- 技術的に深すぎず、一般のエンジニア・AI興味層が理解できる内容
- 注目度が高く、話題性のある記事を優先的に選定

**ターゲット読者の詳細:**
- **メインターゲット:** エンジニア
- 実際にAI/LLMを使って開発している人
- 新しいツール・API・機能を試したい技術者
- 技術的な詳細に興味がある層

### 2. 記事の粒度

**具体的な機能・アップデートにフォーカス:**
- ❌ 一般向けの大きなニュース: 「Claude 4.5リリース」「OpenAI発表」
- ✅ エンジニア向けの具体的な内容: 「Claude 4.5のMCPサポートが強化」「Codexの新API機能追加」
- 実装例・ユースケースが明確なものを選定

### 3. 記事の価値提供（最重要）

**🚨 CRITICAL原則:** ただのニュース発表ではなく、注目度の裏付けと実用性を示す

#### 3-1. バズ度・注目度の数値的証拠（必須）

**必須提示項目（いずれか1つ以上）:**
- Hacker Newsスコア（最低200ポイント以上推奨）
- Redditのupvotes数（最低1000以上推奨）
- GitHub Starの急増（具体的な増加数・期間を記載）
- Twitter/X上でのインプレッション・リツイート数
- ダウンロード数、npm/pip等のパッケージマネージャーの統計

**🚨 重要:** 注目度が数値で証明できないニュースは採用しない

#### 3-2. 開発者の実際の関心事を反映（補足的に使用）

**含めるべき情報:**
- コメント欄での議論内容を要約（数値的裏付けの補足として）
- 「何が便利になったのか」「何が解決されるのか」を具体的に
- ベンチマーク結果、パフォーマンス比較などの実測値

#### 3-3. 実用的な視点を提供

**読者の判断材料として提供すべき情報:**
- 「自分のプロジェクトに使えそうか」の判断材料
- 料金・制限・対応言語などの実務的な情報
- 既存ツールとの比較、移行コストの考察
- 実際の活用シーン・ユースケースの提示

#### 3-4. 記事の良い例・悪い例

**❌ NG例:**
「〇〇社が新機能をリリースしました。便利です」

**✅ OK例:**
「Hacker Newsで1200ポイント獲得した〇〇の新機能。開発者が特に注目しているのは△△の部分で、既存の方法と比べて3倍速いとのベンチマーク結果が出ています」

### 4. 選定基準の詳細

#### 4-1. コンテンツタイプ（優先度順）

1. **最新アップデート** - GPT-5リリース、新モデル発表等
2. **トレンド** - 話題のAI技術・手法・アプリケーション
3. **ニュース** - AI業界の動き、企業発表、規制関連
4. **実用事例** - AIの実際の活用例、成功事例

#### 4-2. ソース優先度

**Tier 1（最優先）:**
- TechCrunch
- The Verge
- Ars Technica
- VentureBeat AI
- AI公式ブログ（OpenAI, Google AI, Anthropic等）

**Tier 2（推奨）:**
- MIT Technology Review
- Wired
- Hacker News（高スコア記事）
- Reddit r/MachineLearning（人気投稿）

**Tier 3（補足）:**
- Medium（AI著名人の記事）
- 企業技術ブログ
- AI系YouTuberの話題

#### 4-3. 注目度指標（必須・数値化必須）

**🚨 CRITICAL原則:** 注目度が数値で証明できないニュースは不採用

**最低基準（以下のいずれか1つ以上を満たす）:**
- Hacker Newsスコア: 200ポイント以上
- Reddit r/MachineLearning upvotes: 1000以上
- Reddit r/programming upvotes: 1000以上
- GitHub Stars: 1週間で+1000以上の急増
- Twitter/X: 主要AI開発者アカウントのツイートで1000インプレッション以上
- npm/PyPI等: 週間ダウンロード数の大幅増加（前週比+50%以上）

**推奨基準:**
- Hacker Newsスコア: 500ポイント以上
- Reddit upvotes: 2000以上
- GitHub Stars: 5000以上または急激な増加トレンド
- 複数のソースで同時に話題になっている

#### 4-4. 記事の新鮮度基準

**必ず2-3日以内（絶対条件・鮮度重視）:**
- **基本ルール:** 2-3日以内の最新ニュース
- **例外条件:** 注目度が非常に高く、二次情報（議論・分析記事）が豊富な場合のみ7日以内まで許可

**題材の多様性:**
- ビッグテック発表だけでなく、エンジニア界隈で徐々に盛り上がりを見せている有望なツール・OSS・フレームワークも積極的に取り上げる
- **開発者ツール優先:** GitHub Trending、Hacker News "Show HN"、Reddit r/programming等で話題の実用ツールは優先度高

#### 4-5. 記事の分かりやすさ基準

**採用基準:**
- 専門用語の適切な説明がある
- 具体例・ユースケースが示されている
- 図表・画像で視覚的に理解しやすい
- 一般エンジニアが読んで理解できるレベル

**除外基準:**
- 数式だらけの学術論文
- 過度に専門的な実装詳細のみ
- 広告・宣伝目的の記事
- 信頼性の低い情報源
- 既に日本語で広く報道済みの話題

## 📝 記事作成ルール

### 1. 人間らしさを重視した記事執筆方針

#### 1-1. 最重要原則

**AIが書いた記事っぽくしない:**
- 機械的な列挙ではなく、会話的なトーン
- 個人的な視点・感想を適度に含める
- 完璧すぎる文章を避け、自然な言い回しを使う
- 絵文字は使用禁止
- 定型文・テンプレート的表現を避ける

#### 1-2. 人間らしい文章の特徴

**自然な表現の要素:**
- 「〜だと思います」「〜かもしれません」など、適度な曖昧さ
- 「個人的には」「興味深いのは」など、書き手の視点
- 「ただし」「一方で」など、自然な接続詞
- 箇条書きばかりでなく、段落で説明
- 完璧な整理ではなく、話の流れを重視

### 2. 絶対に避けるべきAI臭い表現

#### 2-1. AI臭い表現の具体例（使用禁止）

**❌ NG例（機械的な賞賛表現）:**
- 「でもそれこそが価値なんだと思います」← ひねりなしの賞賛的表現は超AI臭い
- 「〜は素晴らしいと思います」
- 「これは画期的です」
- 「非常に重要です」
- 機械的な肯定・称賛だけで終わる文章

#### 2-2. 適切な表現（共感できる軽い懐疑・テクニカルな実体験ベース）

**✅ OK例（技術的視点の実体験ベース）:**
- 「正直、『当たり前じゃん』と思った。でも前にコンテキスト管理を疎かにしてレイテンシ3倍になった経験があるから、体系化は重要かも」
- 「理想論だけど、実際はステートフル管理のコストで詰まりそう」
- 「前に似たアプローチでトークン消費が予想外に増えたから、今回はどうだろう」
- 「便利そうだけど、既存パイプラインへの組み込みコストが気になる」

#### 2-3. 避けるべき強すぎる批判

**⚠️ 使用禁止の批判表現:**
- 「これは使えない」「意味がない」などの断定的否定
- 「こんなの誰が使うんだ」のような攻撃的な表現
- AI業界では強い語気の批判は少ないため、そぐわない
- 共感できない批判、建設的でない批判は不要

#### 2-4. 適切なトーンバランス

**推奨される表現バランス:**
- **軽い疑問・懐疑はOK:** 「本当に実用的かな？」「ハードル高そう」
- **自分の経験からの不安はOK:** 「前に詰まったから心配」
- **期待と不安の両方を書く:** 「期待はあるけど、〇〇が心配」
- **基本は前向き、ただし盲目的な賞賛は避ける**

### 3. 実体験シミュレーションの追加

#### 3-1. 自分事として語る（テクニカルな失敗談）

**✅ 推奨される技術的失敗談:**
- 「前にコンテキスト詰め込みすぎてレイテンシが3倍になった」
- 「同じようなアプローチ試したら、トークン消費が予想外に増えて予算オーバー」
- 「この手法、ステートフル管理が必要で実装コスト高そう」
- 「似たパターンで実装したら、エッジケースの処理で詰まった」

#### 3-2. 避けるべきバカっぽい失敗談

**❌ 使用禁止の低レベル失敗談:**
- 「ドキュメント読んでなくて失敗した」← ただのミス
- 「環境構築で半日溶かした」← 技術的洞察なし
- 「変数名間違えてバグった」← レベルが低い
- 「コピペミスで動かなかった」← 論外

#### 3-3. 技術的な悩み・疑問を入れる

**✅ 推奨される技術的疑問:**
- 「便利そうだけど、既存のパイプラインにどう組み込むか」
- 「理屈は分かるけど、スケールした時のパフォーマンスが心配」
- 「これ、本番環境でのエラーハンドリングどうするんだろう」
- 「メモリ効率とのトレードオフをどう評価すべきか」

### 4. 楽しさ・エンタメ性の追加

**記事に含めるエンタメ要素:**
- 技術的な話にユーモアを混ぜる
- 「あるある」ネタで共感を誘う
- 自虐的な表現も適度に入れる
- 読んでいて「クスッ」となる要素

### 5. 翻訳調・AI翻訳臭さの完全排除

#### 5-1. 翻訳調の特徴（絶対禁止）

**❌ NG: 翻訳で出した日本語のような文章:**
- 欧米人の感情的表現をそのまま直訳したような文章
- 言葉が綺麗すぎて子供の言葉遣いのように聞こえる
- 「〜を与えています」「〜を巻き起こしています」などの硬い動詞表現
- 「特に注目されているのは」「興味深いのは」の多用
- すべての文が同じトーンで書かれている（温度感が一定）

#### 5-2. エンジニアらしいプロフェッショナルな文体

**✅ OK: 基本は淡々とした説明:**
- 無理に注目を引く表現を使わない
- 事実を淡々と述べる（「この研究は〜を示した」程度でOK）
- 不要な修飾語を削る
- 技術用語はそのまま使う（変に言い換えない）

**✅ OK: ポイントで尖った表現を入れる:**
- 本当に重要な部分だけ、エンジニアらしい尖った言い方
- 「これはヤバい」「マジで厄介」など、感情が動いた部分のみ
- 技術的に問題だと思う箇所は、遠慮なく指摘
- ただし、全体の1〜2箇所程度に抑える

#### 5-3. 文体の温度差をつける

**推奨バランス:**
- **9割:** 淡々とした事実説明（「〜である」「〜だ」）
- **1割:** 感情がこもった尖った表現（「これは危ない」「実装コストエグい」）

#### 5-4. 立場・距離感の適切な設定

**❌ NG（当事者意識が強すぎる）:**
- LLM開発者のような当事者意識が強すぎる表現
- 「これはマズい」のような直接的に影響を受ける立場の断定
- 映画の主人公のような劇的・感情的な言い回し
- 「我々開発者にとって」← 直接的な立場を主張
- 「今すぐ対処が必要だ」← 映画的・劇的な表現

**✅ OK（適度な距離感）:**
- 間接的に関係するかもしれない観察者としての適度な関心
- 「業界的には重要そうだから追ってる」程度の距離感
- 「自分には直接関係ないけど、気になる」というスタンス
- 「これはやばいんじゃない？」
- 「直接関係ないけど、業界的には気になる動き」
- 「開発者じゃないから実感ないけど、影響ありそう」
- 「個人的には使わないけど、トレンドとしては追っておきたい」

#### 5-5. 口語的・フランクすぎる表現の回避

**🚨 CRITICAL: Zenn記事にふさわしい表現:**

**❌ NG（フランクすぎる口語表現）:**
- 「〜あるんですけど」「〜なんですけど」← Zenn記事には不適切
- 「〜だったりします」「〜だったりする」← 口語的すぎる

**✅ OK（適度にフォーマルな表現）:**
- 「〜ありますが」「〜ですが」
- 「〜です」「〜します」← 明確な表現

**基本ルール:**
- Zenn記事は技術記事として読まれるため、過度にカジュアルな口語表現は避ける
- 「ですます調」を基本とし、口語的な語尾（「〜けど」「〜だったり」）は使用禁止

#### 5-6. ユーザーの立場・専門性の正確な反映

**🚨 CRITICAL: AIエンジニアとしての適切な立場表現:**

**重要:** ユーザーはAIエンジニアであり、AI業界の動向に間接的に関係がある立場

**❌ NG（専門性を無視した表現）:**
- 「直接関係ない立場なので客観的に見守る」← ユーザーの専門性を無視

**✅ OK（専門性を反映した表現）:**
- 「AIエンジニアとして間接的に影響を受ける可能性がある」
- 「業界の重要な転換点として注視している」
- 「今後の主要IT企業の行く末に影響する可能性がある」

#### 5-7. 翻訳調と自然な表現の比較例

**❌ NG例（翻訳調・子供っぽい）:**
- 「この研究は、AI開発者コミュニティに大きな衝撃を与えています」
- 「特に注目されているのは、その攻撃の『手軽さ』です」
- 「これが意味するのは、『250個の文書を作るのは簡単』という現実です」

**✅ OK例（エンジニアプロ調・適度な距離感）:**
- 「Hacker Newsで563ポイント。それなりに話題になってる」
- 「問題は攻撃が簡単すぎること」
- 「250個なら個人でも作れる。やばいんじゃない？」

### 6. Zenn記事フォーマット

#### 6-1. 基本構造

```markdown
---
title: "[記事タイトル]"
emoji: "📝"
type: "tech"
topics: ["AI", "機械学習", "ニュース"]
published: false
---

## 参照元
[原題] - [ソース名]
[URL]

## [記事ごとに完全に異なる見出しを作成]

[Hacker News/Reddit/Twitter等での反応を具体的数値で提示]
[開発者コミュニティでの注目理由を説明]

## [記事ごとに完全に異なる見出しを作成]

[元記事の要約を会話的に説明。3-4段落程度]

## [記事ごとに完全に異なる見出しを作成]

[コメント欄・議論から見える実際の関心事]
[ベンチマーク・パフォーマンス比較などのデータ]

## [記事ごとに完全に異なる見出しを作成]

[自分のプロジェクトで使えそうか]
[既存ツールとの比較、移行コスト]
[料金・制限・対応言語などの実務情報]

## [実体験を通じて伝えたい結論を一言で表現]

[個人的な感想・考察を含める]

## [懸念点・問題点を具体的に一言で表現]

[技術的・社会的懸念、批判的考察を含める]
```

#### 6-2. 実体験セクション見出しの自然な表現

**🚨 CRITICAL: 実体験を述べることを明示する表現は禁止**

**絶対禁止:**
- 「個人的に〜」「前に〜した経験から言うと」など、実体験を述べることが明示的にわかる表現

**推奨方法:**
- 実体験を通じて最終的に伝えたいメッセージ・結論を見出しにする

**❌ NG例:**
- 「前に音声なし動画で苦労した経験から言うと」
- 「個人的に気になった点」
- 「実際に使ってみて感じたこと」

**✅ OK例:**
- 「音声同期の手間は想像以上に重い」
- 「6秒制限では表現の幅が足りない」
- 「本番環境での信頼性がまだ不安」
- 「生成ガチャ問題が実用化の壁になる」
- 「スケーリングの課題は見過ごせない」
- 「検出技術の遅れが混乱を招く」

**重要原則:**
- 毎回異なる見出しで、「ここで実体験が来るな」とリピーターに予測させない
- 記事の核心的な問題点・懸念点を端的に示す表現を使用

#### 6-3. 懸念点セクション見出しの自然な表現

**🚨 CRITICAL: 懸念を述べることを明示する表現は禁止**

**絶対禁止:**
- 「ここが引っかかる」「懸念点」「気になる点」など、懸念を述べることが明示的にわかる表現

**推奨方法:**
- 懸念点・問題点の具体的な内容を見出しにする

**❌ NG例:**
- 「ここが引っかかる」
- 「懸念事項」
- 「気になるところ」
- 「問題点」

**✅ OK例:**
- 「低品質コンテンツ大量生産のリスク」
- 「動画が証拠にならない時代の到来」
- 「プライバシー保護の代償」
- 「Google製品廃止の前例が不安要素」
- 「スケーリング時のサーバー負荷問題」
- 「初期の新奇性は既に薄れている」

**重要原則:**
- 毎回異なる見出しで、「ここで懸念が来るな」とリピーターに予測させない
- 記事で指摘したい具体的な問題・リスク・課題を端的に示す表現を使用

#### 6-4. 見出しの定型化完全禁止（最重要）

**🚨 CRITICAL: 抽象的・汎用的見出しは使用禁止**

**絶対禁止:**
- 「注目されている理由」「なぜ注目されているのか」
- 「主な内容」「コミュニティでの議論」
- すべての記事で使える抽象的・汎用的表現

**必須:**
- 記事の具体的な内容に基づいた固有の見出しを作成する

**❌ NG例:**
- "注目されている理由" / "なぜ注目されているのか" → 汎用表現
- "主な内容" → 抽象表現

**✅ OK例:**
- "SWE-Bench Proの結果が話題に" → 具体的
- "$100でChatGPTクローンが作れる時代" → 固有の内容
- "攻撃手法の詳細" / "新ベンチマークの仕組み" → 内容を明示

**重要原則:**
- 同じパターンの見出し（"〇〇の理由"、"〇〇の内容"等）を避ける
- 記事タイトルやトピック名を見出しに含めて具体性を出す

#### 6-5. 第一見出しの重要性（読者の興味を引く）

**🎯 第一見出しは記事の価値提案を明確に示す**

**NG例（退屈で汎用的）:**
- "概要"
- "背景"
- "はじめに"

**✅ OK例（具体的で興味を引く）:**
- "$100でChatGPTクローンが作れる時代"
- "GPT-4を超えるベンチマーク結果"

**推奨手法:**
- 数字・具体的な成果・驚きの要素を含める
- 「〜が〇〇になった」「〜で△△を実現」のような具体的表現を使用
- 第一見出しで記事の価値を明確に示す

#### 6-6. 語尾の統一

**必須ルール:**
- 記事本文は「ですます調」で統一
- 「〜だ」「〜である」調は使用禁止
- 自然な会話調を保ちつつ、敬語で統一

1. 必ず `published: false` で作成
2. 元記事のURL・タイトルを明記
3. 著作権・引用ルールを遵守

5. 技術用語は文脈に応じて解説

### 8. 避けるべき表現

**絶対禁止:**
- 「〜について説明します」系の導入
- 「以下のポイントが挙げられます」
- 「〜することができます」の多用
- 完璧に整理された箇条書きのみの構成
- 感情のない機械的な要約
- 「でもそれこそが価値なんだと思います」のような無批判な賞賛
- 「これは素晴らしいです」だけで終わる浅い評価

### 🚨 Zenn記事ファイル名の厳格な制限（CRITICAL）

**絶対に守るべきルール:**

1. **ファイル名（拡張子を除く）= 記事のslug**
   - slug: Zenn記事URLの一部となる識別子
   - 例: `articles/example-article-2025.md` → `https://zenn.dev/tenormusica/articles/example-article-2025`

2. **slug長の厳格な制限: 12〜50文字**
   - 最小: **12文字**
   - 最大: **50文字**
   - **51文字以上はデプロイ中断の原因となる**

3. **使用可能文字: 半角英数字（a-z0-9）、ハイフン（-）、アンダースコア（_）のみ**
   - ✅ OK: `openai-gpt4-release-2025`
   - ✅ OK: `ai_tools_comparison_2025`
   - ❌ NG: `OpenAI-GPT4-Release-2025`（大文字不可）
   - ❌ NG: `openai-gpt4-release-2025-新機能`（日本語不可）
   - ❌ NG: `openai.gpt4.release.2025`（ドット不可）

4. **違反時の影響:**
   - **デプロイが完全に中断される**
   - 他のすべての正常な記事もデプロイされない
   - エラーメッセージ: `articles/[ファイル名].mdはファイル名が不正です。ファイル名（拡張子を除く）は記事のslugとなります。slugは半角英数字（a-z0-9）、ハイフン（-）、アンダースコア（_）の12〜50文字の組み合わせにする必要があります`

5. **記事作成時の必須確認:**
   ```bash
   # ファイル名の文字数確認（拡張子.mdを除く）
   echo "your-article-slug-here" | wc -c
   # 結果が13〜51の範囲（改行含む）であることを確認
   # 実際の文字数は表示値-1（改行除外）
   ```

6. **実際に発生した問題例:**
   - ファイル名: `microsoft-anthropic-claude-partnership-openai-shift-2025.md`
   - slug長: **57文字**（制限: 50文字）
   - **7文字オーバー**でデプロイ全体が中断
   - 影響: 他のすべての正常な記事が公開されず
   - 解決: 不正ファイルを削除後、全記事が正常に公開された
